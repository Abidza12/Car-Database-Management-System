Cars Endpoint & Filtering

The /api/cars endpoint builds SQL dynamically but safely with parameterized queries. It supports filtering by brands (IN list) and features (many-to-many) using subqueries.

@app.route('/api/cars', methods=['GET'])
def get_cars():
    brands_filter = request.args.get('brands', '')
    features_filter = request.args.get('features', '')

    query = '''
        SELECT DISTINCT c.*, b.brand_name,
               GROUP_CONCAT(f.feature_name, ',') as features
        FROM cars c
        JOIN brands b ON c.brand_id = b.brand_id
        LEFT JOIN car_features cf ON c.car_id = cf.car_id
        LEFT JOIN features f ON cf.feature_id = f.feature_id
    '''

    conditions = []
    params = []

    # Brand filtering
    if brands_filter:
        brand_ids = brands_filter.split(',')
        placeholders = ','.join('?' * len(brand_ids))
        conditions.append(f'c.brand_id IN ({placeholders})')
        params.extend(brand_ids)

    # Feature filtering (complex subquery)
    if features_filter:
        feature_ids = features_filter.split(',')
        for feature_id in feature_ids:
            conditions.append(f'''
                c.car_id IN (
                    SELECT car_id FROM car_features WHERE feature_id = ?
                )
            ''')
            params.append(feature_id)

    if conditions:
        query += ' WHERE ' + ' AND '.join(conditions)

    query += ' GROUP BY c.car_id ORDER BY c.name'
    cursor.execute(query, params)

Notes:
• Frontend sends comma-separated IDs for brands and features.
• Backend uses parameterized queries to avoid SQL injection.
• Feature filtering iterates feature IDs and requires the car to match all requested features (AND).

Detailed Technical Explanation of Dynamic Query Building

The dynamic query building in the /api/cars endpoint follows a structured and modular approach, layering filtering logic over a base SQL query. Below is the integrated detailed explanation:

1. Base Query Structure

SELECT DISTINCT c.*, b.brand_name,
       GROUP_CONCAT(f.feature_name, ',') as features
FROM cars c
JOIN brands b ON c.brand_id = b.brand_id
LEFT JOIN car_features cf ON c.car_id = cf.car_id
LEFT JOIN features f ON cf.feature_id = f.feature_id

This query joins cars, brands, and features while ensuring complete rows using LEFT JOIN. DISTINCT prevents duplication due to many-to-many relationships. GROUP_CONCAT aggregates feature names into a single string grouped by car_id.

2. Parameter Extraction

brands_filter = request.args.get('brands', '')
features_filter = request.args.get('features', '')

Filters are extracted as comma‑separated strings from query parameters. Lists conditions and params store SQL fragments and corresponding safe values.

3. Brand Filter Construction

brand_ids = brands_filter.split(',')
placeholders = ','.join('?' * len(brand_ids))
conditions.append(f'c.brand_id IN ({placeholders})')
params.extend(brand_ids)

This creates an IN clause with dynamic placeholders, fully parameterized to avoid SQL injection.

4. Feature Filter Construction (Many‑to‑Many)

for feature_id in feature_ids:
    conditions.append('''
        c.car_id IN (
            SELECT car_id FROM car_features WHERE feature_id = ?
        )
    ''')
    params.append(feature_id)

Each feature requires a separate subquery to ensure cars match ALL selected features (logical AND).

5. Final Query Assembly

if conditions:
    query += ' WHERE ' + ' AND '.join(conditions)
query += ' GROUP BY c.car_id ORDER BY c.name'

The backend appends WHERE only when needed, joins conditions with AND, groups results by car, and orders them alphabetically.

6. Query Execution

cursor.execute(query, params)

The query is executed with safe parameter binding, protecting against injection.

7. Example Generated Queries

No Filters

... GROUP BY c.car_id ORDER BY c.name

Brand Filter: brands=1,2

WHERE c.brand_id IN (?,?)

Brand + Features: brands=1 & features=3,4

WHERE c.brand_id IN (?)
  AND c.car_id IN (SELECT car_id FROM car_features WHERE feature_id = ?)
  AND c.car_id IN (SELECT car_id FROM car_features WHERE feature_id = ?)

8. Security & Performance Considerations

  • Uses parameterized queries to prevent injection
  • DISTINCT, indexing, and optimized joins enhance performance
  • Dynamic query generation scales with filters using O(n) complexity
  • GROUP_CONCAT efficiently aggregates variable‑length feature lists
